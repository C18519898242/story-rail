# StoryRail 最小 CLI 剧情引擎设计

日期：2026-07-19

## 目标

实现一个不依赖 AI、数据库或前端的最小可玩文字冒险闭环，用来验证 StoryRail 的核心设计：玩家只能看到当前场景和可执行动作，剧情引擎根据选项内部定义的固定结果跳转到下一节点，直到抵达结局。

第一版使用一个手写的原创短剧本，包含 5～10 个节点和至少两个结局。

## 本阶段不做什么

- 不调用任何 AI API。
- 不实现 Provider 适配层。
- 不从小说自动生成剧本。
- 不保存或恢复游戏进度。
- 不写 JSONL 游玩记录。
- 不实现人物关系、物品、条件选项或复杂状态变量。
- 不提供 Web、桌面或图形界面。

## 用户体验

玩家在终端启动游戏后，程序读取剧本入口节点，并循环执行：

1. 显示当前场景文字。
2. 按顺序显示玩家可见的固定选项。
3. 等待玩家输入选项编号。
4. 如果输入无效，说明错误并要求重新输入。
5. 显示该选项对应的固定结果文本。
6. 进入选项指定的下一节点。
7. 如果节点是结局，显示结局文本并正常退出。

玩家不会看到节点 ID、内部跳转关系或其他剧本控制字段。

## 最小剧本格式

整个剧本保存在一个 JSON 文件中，顶层包含格式版本、作品信息、入口节点和节点集合。

```json
{
  "schema_version": 1,
  "id": "rainy-night",
  "title": "雨夜相遇",
  "start_node": "rainy_night",
  "nodes": [
    {
      "id": "rainy_night",
      "text": "雨夜里，你发现路边坐着一名受伤的少女。",
      "choices": [
        {
          "id": "help",
          "text": "停下来帮助她",
          "result": "你帮助了少女，她开始信任你。",
          "next": "enter_inn"
        }
      ]
    },
    {
      "id": "safe_ending",
      "text": "天亮后，你们安全离开了小镇。",
      "ending": true
    }
  ]
}
```

### 剧本字段

顶层字段：

- `schema_version`：整数，第一版固定为 `1`。
- `id`：剧本稳定 ID。
- `title`：展示给玩家的标题。
- `start_node`：入口节点 ID。
- `nodes`：节点数组。

普通节点字段：

- `id`：节点稳定 ID，在剧本中必须唯一。
- `text`：当前场景的玩家可见文本。
- `choices`：至少包含一个选项。

选项字段：

- `id`：选项在当前节点中的稳定 ID。
- `text`：玩家可见的行动描述。
- `result`：选择后的固定剧情结果。本阶段直接显示；未来由 AI 将它演绎成对白和描写。
- `next`：下一个节点 ID。

结局节点字段：

- `id`：节点稳定 ID。
- `text`：结局文字。
- `ending`：固定为 `true`。

结局节点不包含 `choices`。普通节点不能设置 `ending: true`。

## 组件边界

### 数据模型

`models.py` 定义剧本、节点和选项的数据结构。数据模型只表达合法数据，不处理终端输入输出。

### 剧本加载和校验

`loader.py` 使用 Python 标准库读取 UTF-8 JSON，并转换为数据模型。加载时执行结构和引用校验：

- 顶层字段完整。
- `schema_version` 受支持。
- 节点 ID 不重复。
- `start_node` 存在。
- 普通节点至少有一个选项。
- 同一节点中的选项 ID 不重复。
- 每个 `next` 都指向存在的节点。
- 结局节点不包含选项。

校验失败时抛出包含具体位置和原因的项目异常，不向调用者暴露难以理解的底层堆栈信息。

### 剧情引擎

`engine.py` 保存当前节点，并提供以下能力：

- 取得当前节点。
- 判断游戏是否结束。
- 根据从零开始的选项索引应用选择。
- 返回所选选项的固定结果。
- 将当前位置更新为 `next` 指定的节点。

引擎不读取 `input()`，不调用 `print()`，也不访问文件，便于独立测试和未来复用。

### CLI

`cli.py` 负责终端交互：显示文字、把玩家输入转换为选项索引、处理无效输入并驱动引擎。业务规则不放在 CLI 中。

第一版通过 Python 模块方式启动：

```bash
python -m storyrail.cli examples/rainy-night/script.json
```

后续再通过 `pyproject.toml` 增加 `storyrail` 命令入口。

## 错误处理

- 文件不存在：显示清楚的文件路径并以非零状态退出。
- JSON 格式错误：显示行号、列号和解析原因。
- 剧本结构或引用错误：列出具体节点、选项和违规字段。
- 玩家输入不是有效编号：不退出游戏，提示后重新输入。
- 运行时遇到不可能的节点引用：作为引擎错误终止，不猜测替代节点。

## 测试策略

测试使用 pytest，测试数据放在 `tests/fixtures/`，不需要网络。

### 加载与校验测试

- 可以加载合法剧本。
- 拒绝重复节点 ID。
- 拒绝不存在的入口节点。
- 拒绝指向不存在节点的选项。
- 拒绝没有选项的普通节点。
- 拒绝包含选项的结局节点。

### 引擎测试

- 从入口节点开始。
- 选择后返回正确固定结果。
- 选择后进入正确节点。
- 无效索引不会改变当前节点。
- 抵达结局后报告游戏结束。

### CLI 测试

- 能从开始节点走到结局。
- 非数字和越界输入会重新询问。
- 玩家输出中不泄露节点 ID 和 `next` 字段。

## 完成标准

- 一个原创的 5～10 节点雨夜故事，至少有两个结局。
- 可以通过 CLI 从入口玩到任一结局。
- 所有无效玩家输入都能恢复。
- 所有无效剧本引用都能在开始游戏前被发现。
- 自动化测试全部通过。
- 实现中不包含 AI、数据库、存档或前端代码。
